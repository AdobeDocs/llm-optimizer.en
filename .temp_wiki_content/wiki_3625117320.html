  • API Key
    • - Not Required
    • For others - Its self service through LLMO UI. Steps documented in each BYOCDN guide https://experienceleague.adobe.com/en/docs/llm-optimizer/using/resources/optimize-at-edge/akamai-byocdn 

  • Allowlist out AI bots
    • Allowlist User Agent (*AdobeEdgeOptimize/1.0*) in the BYOCDN.
    • For CDNs with WAF: For BYOCDNs where there is a firewall (mostly encountered by Akamai - but can be applicable in other CDNs too), and CDN team require us to give more than the user agent string (*AdobeEdgeOptimize/1.0*) . Being an edge service, Optimize at Edge does not exposes any dedicated IPs. Here,  we should avoid giving the IPs, rather we should be following a custom header approach to bypass WAF rules.
      The custom header is: x-edgeoptimize-fetcher-key  that can be set as a hard-to-guess random string in the routing rules. This random string can be generated by any key gen utility available (eg: openssl rand -hex 16, head -c 16 /dev/urandom | xxd -p, https://www.strongdm.com/tools/api-key-generator). This header will be forwarded as is by Tokowaka infra to the BYOCDN.  This header can be used in the WAF rule to allow the Tokowaka requests to go through. The ownership of this key including the generation resides completely with the customer.  This approach can be followed for the lower environments too.

  • Failover
    • Ensure that a fail-over config is there to fallback to original path in case of 4xx/5xx errors from 'Optimize at Edge' Origin. This is a must before we Go-Live. Test this on the dev/stage domain by setting forceFail to true in the site config for the respective domain, and check for the response header x-edgeoptimize-fo: 1

For stage/qa/dev domains

There may be some cases where customers would prefer to test the integration in lower environments before proceeding with prod. These environments are usually protected behind firewall/ip allowlist.

If the Tokowaka infra does not have access to the staging/dev/qa domains, then request the customer to allowlist a custom header in their CDN.

eg: x-edgeoptimize-fetcher-key: <some-secret-value>

This secret value can be generated using any key gen utility available (eg: openssl rand -hex 16, head -c 16 /dev/urandom | xxd -p,https://www.strongdm.com/tools/api-key-generator).

Tokowaka shall forward these headers to the origin as well without logging them.
