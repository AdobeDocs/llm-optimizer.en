Overview

Tokowaka (Optimize at Edge) CDN onboarding falls into three tiers based on who manages the CDN and the level of customer coordination required. This document captures the current state across active customer engagements, recurring friction points, and a prioritized investment roadmap to drive all customers to a predictable onboarding SLA.

Target SLAs:

  • Tier 1 (Adobe Managed): ≤ 1 day
  • Tier 2 (BYOCDN Standard): ≤ 3 days
  • Tier 3 (BYOCDN + WAF): ≤ 7 days

Three-Tier CDN Model

TierCDN TypesWho configuresTarget SLA

CDN Onboarding Challenges

1. CDN Configuration & Routing Mistakes

  • Customers configure at the wrong CDN layer (inner vs outer) — seen in Akamai and Cloudflare cases
  • Wrong CDN type selected in LLMO UI — Thomson Reuters picked BYOCDN when on Adobe Managed Fastly
  • Multi-CDN misclassification — alelo.com.br, eniplenitude.com tagged as AEMCS Fastly but actually on Cloudflare/Akamai; required manual correction

2. Bot Allowlisting Missed Steps

  • AdobeEdgeOptimize/1.0 not allowlisted in BYOCDN → OaE requests blocked post-deployment (MercuryMarine emergency)
  • WAF rules blocking Tokowaka prerender callbacks when x-edgeoptimize-fetcher-key not configured (Merck caching issues after routing was set up)

3. Internal Approval & CDN Team Lag

  • Merck Group (Akamai): CDN routing set aside for weeks due to “multiple priorities”; internal approval required before any rule change
  • GM/Buick (Akamai): staging blocked by header-based auth internal to customer’s CDN config
  • Large enterprises require audit trails and 2-step deployment approval workflows — not yet in the product

4. Stage Environment Restrictions

  • Closed stage environments require IP allowlisting of Tokowaka infra → escalation to engineering
  • Non-apex stage domains cannot self-serve via LLMO UI — engineering must manually onboard

Investment Roadmap

P0 — Prevent Wrong Path (Immediate, low cost)

CDN auto-detection in LLMO UI (LLMO-4091)

  • Auto-detect whether a domain is on AEMCS Fastly, AMS CloudFront, Commerce Fastly, or BYOCDN
  • Surface the correct onboarding path without manual triage
  • Eliminates misclassification errors seen with Thomson Reuters, alelo, eniplenitude

P1 — Close Tier 1 Automation Gaps (6–8 weeks)

Commerce Fastly automation (LLMO-4097, LLMO-4098)

  • Productize the existing Commerce CDN bot for VCL snippet injection
  • Wire into LLMO UI — same flow as AEMCS Fastly

AMS CloudFront automation

  • Script already exists; but need to take care of variety of use cases - it worked for halliburton but not for AARP.
  • Eliminates Eng involvement per customer

P2 — Reduce Akamai Friction (8–12 weeks)

CDN configuration validator (self-serve tool, runs at customer side)

  • Validates headers, cache key, routing, failover config, bot allowlist
  • Catches allowlisting issues before go-live; reduces post-config debugging calls

Akamai PAPI automation (unblock in-progress work)

  • Needs: (a) access to a real Akamai test account, (b) Advanced Behavior contract availability research
  • Cloudflare already has a documented automated path; Akamai is the remaining gap at scale

P3 — WAF Fast-Track (Process investment)

WAF engagement playbook

  • Pre-written security briefing customers hand to their security team (shared secret approach, no IPs, no sensitive data)
  • WAF config templates per CDN
  • Target: reduce WAF security review from “unknown” to 2–3 days

Dedicated WAF track

  • Separate WAF cases in tracking to avoid inflating Tier 2 SLA metrics
  • Assign a dedicated Eng liaison for Tier 3 cases
