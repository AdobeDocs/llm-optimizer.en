Tokowaka Service: Allowing Service Through BYOCDN WAF Rules

Author: Nitin Gupta | Created: 2026-03-26 | Updated: 2026-03-27

32

Overview

The Tokowaka service comprises two components that need to make requests through customer BYOCDN:

  • Tokowaka Edge Worker — runs on Fastly's edge network
  • Tokowaka Prerender Service — runs as an AWS Lambda function

Both services need to be recognized as legitimate callers by customer BYOCDN WAF rules. This document evaluates approaches to allow Tokowaka service through BYOCDN WAF rules and recommends a shared secret header approach over IP-based allowlisting.

Problem Statement

Customers using BYOCDN (Bring Your Own CDN) configure WAF rules that block unrecognized traffic. When Tokowaka service makes requests through customer BYOCDN, these requests are blocked by the WAF. We need a mechanism for BYOCDN to identify and allow Tokowaka requests.

IP allowlisting is particularly problematic because the two Tokowaka components have very different egress characteristics:

ComponentRuntimeIP Allowlisting Feasibility
Edge WorkerFastly edge networkNot viable. VPC/NAT is not supported on edge workers. Would require allowlisting all of Fastly's public IP ranges, effectively providing no WAF protection since any Fastly customer's traffic originates from the same IPs.
Prerender ServiceAWS LambdaPossible but risky. Requires VPC + NAT Gateway for stable EIPs. Limited to 54 usable IPs in current /27 subnets, with hard failures at peak traffic (see analysis below).

A unified solution that works for both components without IP dependency is needed.

Recommended Approach: Shared Secret Header

This is the recommended approach. It eliminates the need for VPC, NAT Gateways, and stable egress IPs entirely.

How It Works

bash Request flow: AI Bot --> BYOCDN --> Tokowaka Service --> BYOCDN --> Customer Origin | (Edge Worker or ^ | Prerender) | | WAF validates | x-edgeoptimize-fetcher-key | and allows request | +-- Routing rules set x-edgeoptimize-fetcher-key on the request to Tokowaka; Tokowaka forwards the header back when calling BYOCDN]]>

An AI bot makes a request to the customer's site via BYOCDN. The BYOCDN routing rules route the request to Tokowaka service (Edge Worker or Prerender) and attach the x-edgeoptimize-fetcher-key header. Tokowaka processes the request and calls back to the same BYOCDN to fetch the page for prerendering, forwarding the header. The BYOCDN WAF validates the header on the callback and allows the request through. The same mechanism works identically for both Tokowaka components.

Key Design: Tokowaka Never Manages the Secret

The secret key is entirely within the customer's BYOCDN configuration. Tokowaka does not see, store, or manage the secret. The BYOCDN routing rules inject the header, and the BYOCDN WAF validates it. Tokowaka simply forwards whatever headers the routing rules attach.

This means:

  • No secrets management on Tokowaka's side — no Secrets Manager, no env vars, no rotation coordination
  • Customer owns the full key lifecycle — generation, rotation, revocation are entirely self-service
  • No risk of Tokowaka leaking the key — it is an opaque passthrough
  • Per-customer by default — each customer sets their own key in their own BYOCDN config

Tokowaka's Responsibility

Tokowaka must ensure that it:

  • Does not strip the x-edgeoptimize-fetcher-key header as it passes through the prerender stack
  • Does not log the header value in any application logs, error messages, or debug output

Customer Configuration Steps

  1. Generate a cryptographically random secret (e.g., openssl rand -hex 32)
  2. Add the secret to their Tokowaka routing rules in BYOCDN configuration as the value for x-edgeoptimize-fetcher-key
  3. Configure a WAF rule: allow requests where x-edgeoptimize-fetcher-key matches the secret (exact match)
  4. To rotate: update both the WAF rule and routing rules with a new key (support two active keys during rotation window if zero-downtime is required)

Evaluated Alternative: IP Allowlisting via VPC + NAT Gateway

Not recommended. This approach was evaluated and found to have significant scalability, cost, and operational risks. Documented here for reference.

This approach places the Lambda inside a VPC with NAT Gateways to provide stable Elastic IPs that CDNs can allowlist. The VPC evaluated is the existing Spacecat architecture VPC (10.0.0.0/25).

Important context: Tokowaka service has no need to access any internal or private network resources. The only reason a VPC was being considered was to obtain stable egress IPs via NAT Gateways for CDN allowlisting. A VPC adds unnecessary complexity, cost, and scalability constraints for a service that exclusively makes outbound calls to public websites. This further strengthens the case for the shared secret header approach.

Scalability Risk

Lambda gets 54 usable IPs across the two private /27 subnets. The /25 VPC is fully allocated with no room to resize.

Observed traffic profile:

MetricValue
Current invocation rate200–300/min
Historical peak invocation rate1,350/min (before VPC was added)
Average duration7 seconds
Max duration20 seconds
Current estimated concurrent~35 (avg), ~100 (peak)
Peak traffic concurrent~158 (avg duration), ~450 (max duration)
Available IPs54 (across 2 private subnets)

At the historical peak of 1,350 invocations/min with average 7s duration, concurrent executions reach ~158. With Hyperplane ENI sharing (3–5 executions per ENI), this requires 32–53 ENIs — at the very edge of the 54 IP limit. At max duration (20s), concurrent reaches ~450, far exceeding available IPs and causing hard invocation failures.

Additional Risks

  • NAT Gateway throughput ceiling: 45 Gbps and ~55,000 simultaneous connections per destination
  • Cold start overhead: ~200–500ms per scale-up event for Hyperplane ENI attachment
  • Cost: ~$65/month base (2 NAT GWs) + $0.045/GB data processing
  • IP change coordination: If NAT GW EIPs ever change, all CDN customers must update their allowlists

VPC Mitigation (If Required for Other Services)

Tokowaka itself has no use case for VPC. However, if the Spacecat VPC is being shared and the IP allowlisting approach were to be used as a fallback, the IP constraint could be mitigated by:

  1. Adding a secondary CIDR block (e.g., 10.0.1.0/24) with larger private subnets — gives ~300 usable IPs
  2. Capping Lambda reserved concurrency at ~150 — converts hard IP-exhaustion failures into graceful throttles

Recommendation

Use the shared secret header approach (x-edgeoptimize-fetcher-key). It eliminates the need for VPC infrastructure, removes all scalability constraints, costs nothing, and puts key management entirely in the customer's hands. The VPC + NAT IP allowlisting approach should only be considered as a fallback for CDNs that exclusively support IP-based allowlisting and cannot configure custom header WAF rules.
